# MySQL 账户存储改造设计

## 目标

将当前基于 `backend/accounts.json` 的账户、任务和项目持久化改为 MySQL 5.7 数据库 CRUD。改造后不再以本地 JSON 文件作为运行时数据源，同时保持现有 Vue 前端接口和响应结构兼容。

## 范围

本次改造包括：

- 账户的查询、新增、修改、删除。
- 账户 Token 的读取与更新。
- 账户任务的查询、新增、修改、删除和排序。
- 远端签到项目列表的读取与更新。
- 调度器从数据库读取最新账户和任务。
- 旧 `backend/accounts.json` 数据的一次性导入。

本次不改造以下存储：

- `settings.json`
- `users.json`
- `sessions.json`

## 数据库连接

目标数据库为本机 MySQL 5.7.44：

- 主机：`127.0.0.1`
- 端口：`3306`
- 数据库：`xxqd`
- 用户：`root`

连接配置从环境变量读取。开发环境可使用用户提供的密码作为未配置环境变量时的默认值，但密码不得写入受 Git 跟踪的配置文件、日志或文档。应用启动时先连接 MySQL 服务；若 `xxqd` 不存在则创建，再连接目标数据库并创建缺失的数据表。

建议的环境变量：

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`

数据库连接失败时应用启动失败，并输出不包含密码的明确错误信息。系统不回退到 `accounts.json`。

## 架构

新增独立数据库模块和仓储层：

1. 数据库模块负责连接、会话、建库和建表。
2. ORM 模型负责关系表映射。
3. 账户仓储负责账户、任务、项目的数据库 CRUD 和字典结构转换。
4. `AppState` 保留签到、通知、调度和日志等业务职责，通过仓储访问数据。

`AppState` 不再长期维护 `self.accounts` 作为权威数据源，也不再调用整文件读写函数。快照、接口请求、Token 刷新和调度循环均从仓储读取当前数据库状态。

现有 API 路径继续使用 `account_index` 和 `task_index`。仓储按稳定顺序查询记录，并将列表索引解析为数据库主键，以保持前端兼容。数据库内部关联使用主键，不使用列表索引作为外键。

## 数据模型

### accounts

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | BIGINT | 主键，自增 |
| `name` | VARCHAR(255) | 非空 |
| `mobile` | VARCHAR(64) | 非空，唯一索引 |
| `password` | VARCHAR(255) | 非空 |
| `token` | TEXT | 非空，默认空字符串 |
| `created_at` | DATETIME | 非空 |
| `updated_at` | DATETIME | 非空 |

远端签到要求应用能够取回原始密码，因此本次保持密码明文存储，与现有行为一致。密码不得输出到日志。

### tasks

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | BIGINT | 主键，自增 |
| `account_id` | BIGINT | 外键，非空，级联删除 |
| `position` | INT | 非空，保存显示顺序 |
| `project_index` | INT | 非空，对应当前任务的 `index` |
| `title` | VARCHAR(255) | 非空 |
| `times` | JSON | 非空 |
| `enable` | BOOLEAN | 非空 |
| `use_location` | BOOLEAN | 非空 |
| `text` | TEXT | 非空 |
| `pic_paths` | JSON | 非空 |
| `skip_weekends` | BOOLEAN | 非空 |
| `mode` | VARCHAR(16) | 非空 |
| `notify_wechat` | BOOLEAN | 非空 |

`account_id + position` 建立唯一约束。任务增删后在同一事务中重新整理位置，保证 API 索引连续。

### account_projects

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | BIGINT | 主键，自增 |
| `account_id` | BIGINT | 外键，非空，级联删除 |
| `position` | INT | 非空 |
| `payload` | JSON | 非空，保存远端项目完整结构 |

`account_id + position` 建立唯一约束。刷新项目时，在单个事务内替换指定账户的项目列表。

## 数据流

### 查询

仓储按账户主键升序读取账户，并按 `position` 读取其任务和项目，转换为当前 API 所需的嵌套字典。响应中不暴露数据库主键，以避免前端改动。

### 新增与修改

输入数据继续经过现有标准化和校验逻辑，再由仓储写入对应行。账户修改只更新账户字段，不覆盖任务。手机号重复时返回明确业务错误。

### 删除

删除账户时由数据库外键级联删除任务和项目。删除单个任务后，在同一事务内重排剩余任务的 `position`。

### Token 与项目

登录或刷新 Token 成功后，仅更新目标账户的 `token` 列。获取远端项目成功后，仅替换目标账户的项目行。

### 调度

调度循环每轮通过仓储读取最新账户和任务快照，再执行当前调度判断。后台运行期间通过 API 完成的 CRUD 无需重启即可生效。

## 旧数据迁移

应用启动并完成建表后执行以下检查：

1. 若 `accounts` 表已有记录，不执行迁移。
2. 若 `accounts` 表为空且 `backend/accounts.json` 存在，读取并标准化旧数据。
3. 在单个事务中导入账户、任务和项目。
4. 导入失败则整体回滚，并让应用启动失败。
5. 导入成功后保留原 JSON 文件作为备份，但运行时不再读取或写入它。

该策略保证迁移幂等，不重复导入数据。

## 事务与错误处理

- 每次业务写操作使用独立数据库事务。
- 任一写入失败时回滚整个操作。
- 账户或任务索引越界继续转换为当前 API 的业务错误格式。
- 唯一手机号冲突转换为可读的业务错误，不向客户端暴露 SQL。
- 数据库连接或查询异常写入服务日志，但连接密码和账户密码必须脱敏。
- 不使用 JSON 文件作为数据库故障时的降级写入目标。

## 测试策略

实施遵循测试驱动开发，先编写并运行失败测试，再实现最小代码使其通过。

测试覆盖：

- 数据库初始化和缺失数据库创建。
- 旧 JSON 仅在空库中导入一次。
- 账户增删改查和手机号唯一性。
- 任务增删改查、顺序重排及账户级联删除。
- Token 更新和项目列表替换。
- API 响应结构和现有索引式路径兼容。
- 调度器读取数据库中的最新状态。
- MySQL 5.7 实际连接、JSON 字段和外键行为。

MySQL 集成测试创建名称带随机后缀的唯一临时数据库，例如 `xxqd_test_<随机值>`。测试只清理本次创建的临时数据库，绝不修改或删除正式数据库 `xxqd`。

## 验收标准

- 代码中不再通过 `accounts.json` 执行运行时 CRUD。
- 前端无须修改即可完成现有账户和任务操作。
- 数据在后端重启后仍从 MySQL 正确读取。
- 数据库中的外键、唯一性和任务顺序约束生效。
- 旧 JSON 数据可一次性导入，且不会重复导入。
- 自动化测试、后端导入检查和前端构建均通过。
