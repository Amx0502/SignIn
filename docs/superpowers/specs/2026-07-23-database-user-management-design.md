# 数据库认证与用户管理设计

## 目标

将后台登录用户和会话从 `users.json`、`sessions.json` 迁移到独立的 MySQL 数据库 `User`，并在现有管理后台增加管理员专用的一级“用户管理”菜单和页面。

现有签到业务数据库 `xxqd` 保持独立，不改变账户、任务和项目的数据模型。

## 范围

本次包括：

- MySQL `User` 数据库初始化。
- 后台用户与会话的数据库存储。
- 登录、退出和 Token 验证改为数据库查询。
- 旧 `users.json` 一次性迁移。
- 密码哈希升级。
- 管理员用户 CRUD、密码重置和状态管理接口。
- 用户管理页面、一级侧边栏菜单和前端权限守卫。
- 初始密码或重置密码后的强制改密流程。

本次不包括：

- 签到业务账号的角色或权限划分。
- 邮箱字段。
- 基于组织、部门或细粒度权限点的授权模型。

## 双数据库架构

### `xxqd`

继续存放：

- `accounts`
- `tasks`
- `account_projects`

### `User`

新增：

- `users`
- `user_sessions`

认证数据库使用独立环境变量：

- `AUTH_DB_HOST`，默认 `127.0.0.1`
- `AUTH_DB_PORT`，默认 `3306`
- `AUTH_DB_NAME`，默认 `User`
- `AUTH_DB_USER`，默认 `root`
- `AUTH_DB_PASSWORD`

应用启动时分别初始化 `xxqd` 和 `User`。任意数据库初始化失败，服务启动失败，不回退到本地认证文件。

## 数据表

### users

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | BIGINT | 主键，自增 |
| `username` | VARCHAR(50) | 非空，唯一索引 |
| `password_hash` | VARCHAR(255) | 非空 |
| `role` | VARCHAR(16) | 非空，仅 `admin` 或 `user` |
| `is_active` | BOOLEAN | 非空 |
| `must_change_password` | BOOLEAN | 非空 |
| `created_at` | DATETIME | 非空 |
| `updated_at` | DATETIME | 非空 |
| `last_login` | DATETIME | 可空 |

不保存邮箱。

### user_sessions

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | BIGINT | 主键，自增 |
| `user_id` | BIGINT | 外键，非空，级联删除 |
| `token_hash` | CHAR(64) | 非空，唯一索引 |
| `ip_address` | VARCHAR(64) | 非空 |
| `created_at` | DATETIME | 非空 |
| `expires_at` | DATETIME | 非空 |

客户端持有随机原始 Token，数据库只保存 Token 的 SHA-256 摘要。

## 密码安全

新密码使用 Python 标准库 PBKDF2-HMAC-SHA256：

- 每个密码使用独立随机盐。
- 哈希包含算法标识、迭代次数、盐和摘要。
- 登录时使用常量时间比较。
- 日志和 API 响应不得包含密码或密码哈希。

旧 `users.json` 中的 SHA-256 加盐哈希允许首次登录校验。首次校验成功后，在同一事务中升级为 PBKDF2。

## 数据迁移与初始管理员

应用启动并完成建表后：

1. `users` 表已有数据时不执行迁移或默认用户创建。
2. `users` 表为空且 `users.json` 存在时，一次性导入旧用户。
3. 导入时忽略邮箱，为旧用户设置 `admin` 角色并保持启用状态。
4. 导入失败时整体回滚并终止启动。
5. 导入成功后保留原文件作为备份，运行时不再读写它。
6. 数据库和旧文件都没有用户时创建 `admin / admin123`。
7. 默认管理员标记 `must_change_password=true`。

旧 `sessions.json` 不迁移，部署后所有用户重新登录。运行时不再读写该文件。

## 认证数据流

### 登录

1. 根据用户名和 IP 检查内存中的失败次数与锁定状态。
2. 从 `User.users` 查询用户。
3. 统一以“用户名或密码错误”响应不存在用户和密码错误。
4. 检查用户是否启用。
5. 校验 PBKDF2 或兼容旧哈希，并在需要时升级哈希。
6. 生成随机 Token，将摘要写入 `user_sessions`。
7. 更新 `last_login`。
8. 返回用户名、角色、必须改密标记、Token 和过期时间，不返回邮箱。

连续失败 5 次后，用户名和 IP 组合锁定 5 分钟。锁定记录只保存在进程内存。

### Token 验证

每次受保护请求：

1. 计算 Token 摘要。
2. 联表查询会话与用户。
3. 检查会话未过期、用户启用。
4. 除改密与退出接口外，`must_change_password=true` 的用户不得访问其他业务接口。
5. 过期会话及时删除。

### 退出

删除当前 Token 对应的会话。

## 权限模型

- `admin`：可使用签到功能和用户管理功能。
- `user`：可使用签到功能，不能查看用户管理菜单或调用用户管理接口。

安全约束：

- 系统始终至少保留一个启用中的管理员。
- 禁止删除、禁用或降级最后一个启用中的管理员。
- 管理员不能删除自己。
- 禁用或删除用户时删除其全部会话。
- 重置他人密码时删除目标用户全部会话。
- 重置自己的密码时保留当前会话、删除其他会话。

后端是最终权限边界；前端隐藏菜单和路由守卫只用于用户体验。

## API

### 认证

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `POST /api/auth/verify`
- `POST /api/auth/change-password`

改密接口要求当前密码和新密码。成功后清除 `must_change_password`。

### 用户管理

- `GET /api/users`
- `POST /api/users`
- `PUT /api/users/{user_id}`
- `POST /api/users/{user_id}/reset-password`
- `DELETE /api/users/{user_id}`

用户管理接口仅允许 `admin`。

新增用户输入：

- `username`
- `password`
- `role`
- `is_active`

编辑用户输入：

- `username`
- `role`
- `is_active`

密码重置输入：

- `new_password`

## 前端

### 侧边栏

新增一级菜单“用户管理”，与“小小签到”并列，不放入签到子菜单。仅当当前登录用户角色为 `admin` 时显示。

### 用户管理页面

路由为 `/users`，包含：

- 用户名
- 角色
- 状态
- 最后登录时间
- 创建时间
- 新增、编辑、重置密码、删除操作

页面沿用现有 Element Plus 管理页面风格，使用表格、状态标签、确认框和表单对话框，并适配窄屏。

### 强制改密

当登录响应中的 `must_change_password=true`：

- 跳转 `/change-password`。
- 除退出外不能进入其他页面。
- 改密成功后刷新本地用户信息并进入系统概览。

### 路由守卫

- 未登录用户跳转登录页。
- 普通用户访问 `/users` 跳转系统概览。
- 必须改密用户访问其他页面跳转改密页。
- 后端返回 401 时清理本地会话并跳转登录页。
- 后端返回 403 时提示无权限并跳转允许访问的页面。

## 事务与错误处理

- 用户新增、修改、删除、密码重置和会话清理使用事务。
- 用户名重复返回 HTTP 400 业务错误，不暴露 SQL。
- 未认证返回 401，无管理员权限返回 403。
- 最后一个管理员保护返回明确的 HTTP 400 业务错误。
- 登录失败不区分用户名不存在和密码错误。
- 认证数据库异常不回退到 JSON 文件。

## 测试

后端测试使用唯一临时认证数据库，不修改正式 `User`。

覆盖：

- `User` 数据库和两张表自动创建。
- `users.json` 一次性迁移、回滚和默认管理员创建。
- PBKDF2 哈希、旧哈希校验和自动升级。
- 登录、退出、Token 验证、会话过期和用户禁用。
- 用户 CRUD、密码重置和会话失效。
- 最后一个管理员保护。
- 普通用户调用管理接口返回 403。
- 登录和验证响应不包含邮箱。
- 强制改密访问控制。
- 前端管理员菜单、普通用户路由拦截和用户管理页面构建。

## 验收标准

- 登录用户和会话只从 `User` 数据库读写。
- `users.json`、`sessions.json` 不再用于运行时认证。
- 管理员可从一级“用户管理”页面完整管理用户。
- 普通用户无法查看或调用用户管理功能。
- 默认或重置密码用户必须先改密。
- 正式 `User` 数据库可创建并完成旧用户迁移。
- 后端测试和前端生产构建通过。
